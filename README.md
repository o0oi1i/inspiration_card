# 灵动卡片

## 项目简介
       基于Flutter构建的交互式卡片组件。将《山海经》与现代赛博朋克视觉风格相结合，通过手势拖动卡片，实现半透明灵动卡片堆叠交互效果。

## 功能特性
1. 暗色调色彩渐变模糊透明卡片，增加卡片质感。
2. 随拖动卡片手势移动背景繁星，加深层次感。
3. 随拖动卡片手势卡片增加凝聚背景光效，旋转卡片角度，增强视觉中心感。

## 设计与实现手记
### 1. 技术选型
#### 1.1 状态管理：
Flutter原生状态管理setState来实现该项目，通过监听手势位置变化使用setState进行状态变化，确保手势与动画衔接的流畅性，项目数据量和交互逻辑并不十分复杂未采用Bloc等较为复杂的状态管理框架。

#### 1.2 目录架构
项目采用模块化的目录结构，确保代码的可读性与扩展性：
models：模型层，如卡片名称、配色等。
views：视图层，卡片与氛围背景效果绘制，方便独立调试优化视觉表现。
controllers：控制层，处理手势回调与动画，计算手势拖动和停止时状态数据并将最终状态数据反馈给视图层进行显示。

### 2. 核心实现
主要利用GestureDetector监听手势拖动和释放事件，实时获取水平方向拖动的距离，根据其与阈值（1/4屏幕宽度）判断是否触发卡片飞出或弹回效果；并利用AnimationController控制卡片飞出或弹回效果；卡片，光圈以及背景的动效则根据最上层卡片的颜色和在水平方向上拖动的距离以及与阈值的比例来实时动态的分别设置它们的背景颜色、透明度或旋转角度。

### 3. 权衡与展望
#### 3.1 当前的局限性
* 数据耦合：目前数据硬编码在代码中，未接入外部持久化存储。
* 布局固定：卡片尺寸固定，还需适配多种屏幕尺寸。
* 性能优化：目前未用DevTools监测运行时CPU和内存相关状态。

#### 3.2 未来高并发和复杂度的重构思路
当交互逻辑变得复杂，如增加点击翻转、长按详情等时，可以引入Bloc使用MVVM模式，将业务逻辑与UI分离，表示层负责UI呈现，领域层负责业务逻辑交互和状态管理，数据层负责原始数据存储与获取。
当卡片数量很多时，为避免获取数据图片等的性能问题，可以采用分页加载的方式，仅加载当前可见区域的卡片，并利用isolate并发加载后续几页数据，避免阻塞主线程。
